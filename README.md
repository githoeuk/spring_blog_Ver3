# 🚀 성능 최적화 및 안정성을 고려한 블로그 플랫폼 (Spring Blog Ver.3)

<br>

## 1. 프로젝트 개요
- **진행 기간:** 202X.XX.XX ~ 202X.XX.XX (기간 수정 필요)
- **개발 인원:** 1인 풀사이클 개발
- **기획 의도:** 대용량 트래픽 환경에서도 무너지지 않는 안정적인 백엔드 아키텍처 설계와 객체지향적인 코드 작성을 목표로 한 블로그 웹 서비스입니다. 기능 구현을 넘어 **DB 커넥션 관리, 쿼리 최적화, 전역 예외 처리** 등 서버의 병목 현상을 선제적으로 해결하는 데 집중했습니다.

<br>

## 2. 사용 기술 (Tech Stack)
- **Backend:** Java, Spring Boot, Spring Data JPA
- **Database:** MySQL
- **Tools:** GitHub, IntelliJ IDEA

<br>

## 3. 핵심 아키텍처 및 구현 기능
- **MVC 패턴 기반 REST API 설계:** 사용자 인증, 게시글 CRUD, 댓글 처리 로직 구현
- **보안 및 무결성 확보:** 엔티티(Entity)의 외부 노출을 막기 위해 요청/응답 DTO 철저히 분리
- **AOP 및 Interceptor 적용:** 중복 코드 제거 및 비즈니스 로직 분리

<br>

## 4. 💡 트러블 슈팅 및 성능 최적화 (Key Point)

### ① 불필요한 DB 커넥션 방지 (OSIV 비활성화)
- **Problem:** 기본적으로 켜져 있는 `OSIV(Open Session In View)` 옵션으로 인해, API 응답이 끝날 때까지 DB 커넥션을 물고 있어 트래픽 증가 시 병목 현상이 발생할 위험 발견.
- **Solution:** `application.yml`에서 OSIV를 `false`로 설정하여 트랜잭션이 끝나는 즉시 DB 커넥션을 반환하도록 아키텍처 개선.

### ② 지연 로딩(Lazy Loading) 예외 및 N+1 문제 해결
- **Problem:** OSIV 비활성화 이후, View 단에서 연관된 엔티티를 호출할 때 프록시 초기화 불가로 인한 지연 로딩 예외 발생.
- **Solution:** JpaRepository에서 `Fetch Join`을 적극 도입하여 연관된 데이터를 한 번의 쿼리로 미리 가져오도록 설계. 이를 통해 지연 로딩 예외를 방지하고 N+1 쿼리 문제를 선제적으로 차단하여 데이터 조회 성능 대폭 최적화.

### ③ AOP 기반 전역 예외 처리 및 세션 인증 공통화
- **Problem:** 컨트롤러마다 반복적으로 작성되는 예외 처리와 세션 검증 로직으로 인해 코드 가독성이 떨어지고 유지보수가 어려움.
- **Solution:** - `@ControllerAdvice`를 활용하여 애플리케이션 전역에서 발생하는 예외를 한 곳에서 처리하는 AOP 기반 예외 처리 구현.
  - `HandlerInterceptor`를 통해 로그인 세션 인증 로직을 공통화하여 핵심 비즈니스 로직과 부가 기능을 완벽하게 분리.
